CTF Writeup: TakeOver (Subdomain Enumeration)

November 11, 2025 7 min read CTF | Subdomain Enumeration

1. Overview & Goal

The "TakeOver" challenge revolves entirely around the concept of subdomain enumeration. The scenario involves finding a vulnerability that "blackhat hackers" could use to compromise the company, futurevera.thm.

  • Target Hostname: futurevera.thm
  • Target IP: 10.201.86.245
  • High-level Goal: Discover hidden subdomains/virtual hosts to exploit a misconfiguration leading to information disclosure.

2. Initial Foothold / Enumeration

The hint explicitly directs us to modify /etc/hosts, confirming the use of Virtual Hosts for subdomain mapping. The main Nmap scan confirms services are running on ports 80/443 on the target IP.

Port Scanning (Nmap)

/etc/hosts Configuration

The first mandatory step to access the main website:

10.201.86.245 futurevera.thm

3. Core Vulnerability: Subdomain Enumeration & Error Disclosure

The website text mentioning "blog" and "support" implies the existence of subdomains. We perform **Virtual Host Fuzzing** using **FFuf** by modifying the `Host` header to uncover hidden sites sharing the same IP.

FFuf Execution (Virtual Host Fuzzing)

Running the scan against the IP with the host header fuzzed:

:: Header       : Host: FUZZ.futurevera.thm
:: Wordlist     : FUZZ:/usr/share/wordlists/dirb/common.txt
...
blog     [Status: 200, Size: 3838, Words: 1326, Lines: 81, Duration: 241ms]
Support  [Status: 200, Size: 1522, Words: 367, Lines: 34, Duration: 239ms]
...

The scan confirms two new subdomains: blog.futurevera.thm and support.futurevera.thm. We add these to /etc/hosts.

4. Exploitation: AWS S3 Bucket Leakage

Navigating to support.futurevera.thm triggered a generic, unhandled connection error that became the key to exploitation.

Error Message Analysis:

The error message inadvertently revealed the full, intended destination for the resource, which was an AWS S3 bucket name. This is a severe Information Disclosure flaw:

We can't connect to the server at flag{bee0a0d6edfc...}.s3-website-us-west-3.amazonaws.com.

The structure of the S3 URL contains the flag itself. The vulnerability is a Misconfiguration where internal asset names (which should be non-sensitive) were exposed in the error logging.

5. Remediation / Lessons Learned

The primary lesson is that **verbose errors are often critical vulnerabilities**:

  • Clean Error Handling: All unhandled server and infrastructure errors (especially in cloud environments) must be caught and replaced with generic, non-informative pages (e.g., HTTP 500).
  • Asset Hygiene: Internal asset names (like S3 bucket names) should **never** contain sensitive data, as they are often leaked via verbose errors, logs, or DNS configurations.
  • S3 Bucket Security: Ensure CNAME/DNS records are correctly managed and that S3 bucket policies restrict public access unless strictly necessary.

Conclusion

The challenge was solved by successfully performing **subdomain enumeration (Virtual Host Fuzzing)** and exploiting an **Information Disclosure** vulnerability via a leaked AWS S3 bucket name in a connection error.
